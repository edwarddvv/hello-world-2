### Outline why developers use version control ###

Version control has several advantages for code development.  Every commit is saved, so if a mistake is made e.g. a file is deleted which is later required again, the original code is not lost.  The programmer can go back in the history to the point just before the file was deleted, retrieve the file and then add it back to the latest version of the project (with any modification necessary to get it to run with the current version of the code).  Apart from the ability to correct mistakes, the commit history gives is a record of how the project evolved and the reasons for changes and additions that were made to the code.  This is very useful after significant time has elapsed and the developers have forgotten the reasons for changes (or they have left the company).  Branching helps keep work on the project compartmentalized: different developers can make changes to different parts of the code without interfering with each other.  Branches also allow developers to work on the code without affecting the main branch, so users can continue to use the existing (stable) code until the new code is ready to merge with it.  The version control server is almost always stored on a separate computer to the one on which the coding occurs (and this is definitely true if a external provider like *GitHub* is used) and so the code is present on multiple computers and the server computer will have automatic backups.  Therefore, if the local computer's disk crashes, the work isn't lost.

### Explain ways to use version control in a small Python project ###

A good workflow when implementing a new feature in a python project is to create a new branch in which to work.  Each change should be committed in the branch before starting on the new change, so that changes can be tracked or undone and so that someone reading the commit history will have a clear idea of what changes were made.  e.g. if you want to add an extra parameter to a function then all the files containing either the function implementation or function call should be modified and then the change should be committed with a descriptive message.  Other changes should not be made until this the commit has been made.  This is managed with *git add*.  Only files that have been added to the staging area will be committed, so only run *git add* on files that contain the changes for this commit.  The commits should be regularly pushed to the remote server for backup purposes.  When the feature has been completed and tested, a pull request should be issued so that collaborators can decide whether they agree with the changes and if so, the branch can be merged into the original branch (probably main).  Finally, the branch should be deleted so that the project is less cluttered.
